# RFC-0005: Over-The-Air (OTA) Firmware Updates

- **Status**: Implemented
- **Author**: Agent
- **Date**: 2026-04-12

## 1. Problem

Flashing firmware requires physical access to the device via USB. For remote
deployments the user has no way to update the firmware without travelling to
the device.

## 2. Design

### 2.1 Dual-Partition Layout

ESP-IDF's native OTA mechanism uses two app partitions. The bootloader always
boots the "active" one. An OTA update writes to the "inactive" slot, then
marks it active and reboots.

```
partitions_ota.csv  (4 MB flash)
┌──────────┬─────────┬──────────────┐
│ nvs      │ 24 KB   │ 0x009000     │
│ otadata  │  8 KB   │ 0x00F000     │
│ phy_init │  4 KB   │ 0x011000     │
│ ota_0    │ ~1.94 MB│ 0x020000     │
│ ota_1    │ ~1.94 MB│ 0x210000     │
└──────────┴─────────┴──────────────┘
```

### 2.2 Delivery Channel

HTTPS pull from a user-configured URL (typically a GitHub Releases asset).
The device connects via WiFi, sends an HTTP GET, and streams the response
body directly into the inactive OTA partition — no full-image buffering.

GitHub Releases URLs redirect (302). The implementation follows up to 3
redirects before the final GET.

### 2.3 Configuration

```toml
[ota]
url     = "https://github.com/user/repo/releases/latest/download/smsgate.bin"
confirm = "auto"   # or "manual"
```

- **url** — empty string disables OTA entirely.
- **confirm** — `"auto"`: mark valid on boot; `"manual"`: wait for `/update confirm`.

### 2.4 Boot Confirmation and Rollback

Enabled via `CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE=y` in sdkconfig.

| Mode | Behaviour |
|------|-----------|
| `auto` | `mark_running_slot_valid()` is called early in boot, before "smsgate ready". If the new firmware panics before that point, the next reboot will roll back. |
| `manual` | The user must send `/update confirm` within the rollback window. If they don't (or the device reboots for any reason), the bootloader reverts to the previous slot. |

### 2.5 Bot Command

`/update` replaces the removed `/queue` command (slot 10 of 10).

| Invocation | Action |
|------------|--------|
| `/update` | Download firmware from configured URL, flash, reboot |
| `/update confirm` | Mark current firmware valid (manual mode only) |

The download+flash is a blocking operation that runs in the main loop after
the poller delivers the `UPDATE_SENTINEL`. Progress is logged every 128 KB.

## 3. Constraints

- **Flash budget**: Each OTA slot is ~1.94 MB. Current release binary is
  ~1.3 MB with TLS bundle — leaves ~640 KB headroom.
- **RAM**: The download streams in 4 KB chunks; no full-image buffer.
- **WiFi required**: OTA uses the WiFi connection, not the cellular modem.
- **Single concurrent update**: `EspOta` is a singleton; only one update
  at a time.

## 5. What Was NOT Done

- **Delta updates** — full image only; delta requires server-side diffing
  infrastructure.
- **Telegram-delivered firmware** — too fragile (file size limits, encoding).
- **Automatic periodic checks** — user triggers manually via `/update`.
  Periodic polling could be added later if desired.
- **Version comparison** — the device downloads unconditionally. A future
  enhancement could compare `esp_app_desc_t.version` before downloading.
